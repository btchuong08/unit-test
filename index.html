Our Target: Better Code!

What is Unit Testing? (The Core Idea)

  • Testing Small, Isolated Parts: Focus on the smallest testable piece of code (a function, a method).
  • Independent: Each test should run independently.
  • Fast: Unit tests should execute quickly.
  • Focus: Verify a single piece of logic works as expected.

Breaking down the big problem

Why Unit Test? The Benefits for Us

  • Early Bug Detection: Catch issues before they hit production.
  • Refactoring Confidence: Change code without fear of breaking existing functionality.
  • Improved Code Quality: Forces better design (e.g., pure functions, clear responsibilities).
  • Living Documentation: Tests describe how code *should* behave.
  • Faster Feedback Loop: No more lengthy manual testing.
  • Team Confidence: Deploy with greater assurance.

Our First Test: `sum.js`

// src/sum.js
export const sum = (a, b) => a + b;

// Using CommonJS is also valid
// module.exports = sum;

// __tests__/sum.test.js
import { sum } from '../src/sum';

test('adds 1 + 2 to equal 3', () => {
  expect(sum(1, 2)).toBe(3);
});
  • `test()` or `it()`: Defines a test case.
  • `expect()`: The value you want to test.
  • `toBe()`: A "matcher" for strict equality.

Running Tests & Basic Assertions

npm test
# or
npx jest

// Modern Jest matchers
expect(value).toEqual(anotherValue); // Deep equality for objects/arrays
expect(value).not.toBeNull();
expect(value).toBeUndefined();
expect(value).toBeTruthy();
expect(value).toBeFalsy();
expect(array).toContain(item);
expect(object).toHaveProperty('key.nested');
expect(string).toMatch(/regex/);
expect(() => fn()).toThrow();

// Newer Jest matchers
expect(fn).toHaveReturned(); // Was the function successfully returned from?
expect(mockFn).toHaveBeenCalledOnce(); // Was it called exactly once?
expect(value).toStrictEqual(other); // Stricter than toEqual (no type conversion)
expect(array).toContainEqual({id: 1}); // Contains object with equal fields

Jest provides a rich set of matchers.

Jest Expect API

Organizing Tests with `describe`

// __tests__/calculator.test.js
import { add, subtract } from '../src/calculator';

describe('Calculator operations', () => {
  test('add function should correctly sum two numbers', () => {
    expect(add(5, 3)).toBe(8);
  });

  it('subtract function should correctly subtract two numbers', () => {
    expect(subtract(10, 4)).toBe(6);
  });
});

`describe()` groups related tests, making output clearer.

Setup & Teardown: `beforeEach`, `afterEach`

describe('User management', () => {
  let users = [];

  beforeEach(() => {
    // Runs before *each* test in this describe block
    users = ['Alice', 'Bob'];
    console.log('Setup for a test');
  });

  afterEach(() => {
    // Runs after *each* test
    users = [];
    console.log('Cleanup after a test');
  });

  test('should add a new user', () => {
    users.push('Charlie');
    expect(users).toContain('Charlie');
    expect(users).toHaveLength(3);
  });
});
  • `beforeAll`/`afterAll`: Once for all tests in `describe`.
  • `beforeEach`/`afterEach`: Before/after *each* test.
  • Ideal for resetting state or setting up shared resources.

Testing Asynchronous Code

// src/fetchData.js
export const fetchData = () => {
  return new Promise(resolve => {
    setTimeout(() => resolve('peanut butter'), 100);
  });
};

export const fetchDataThatFails = () => {
  return new Promise((_, reject) => {
    setTimeout(() => reject(new Error('error fetching data')), 100);
  });
};

// Modern async testing approaches
import { fetchData, fetchDataThatFails } from '../src/fetchData';

// Approach 1: Using async/await (clean and readable)
test('the data is peanut butter', async () => {
  const data = await fetchData();
  expect(data).toBe('peanut butter');
});

// Approach 2: Using resolves/rejects matchers (even cleaner)
test('data promise resolves to peanut butter', () => {
  return expect(fetchData()).resolves.toBe('peanut butter');
});

test('fetching data fails with error', () => {
  return expect(fetchDataThatFails()).rejects.toThrow('error fetching data');
});

Jest handles Promises and `async/await` gracefully.

Mocking: The Key to Isolation

  • Why Mock? Isolate the "unit under test" from its dependencies (e.g., database, external APIs, file system).
  • Control Behavior: Simulate different scenarios (success, error, specific return values).
  • Speed: Avoid slow external calls.

Analogy: A stunt double in a movie. The main actor (your unit) performs, but the dangerous/complex parts are handled by a substitute (mock).

`jest.fn()`: Mocking Functions

// Modern mocking syntax
const mockCallback = jest.fn(x => 42 + x);

mockCallback(0); // Call it
mockCallback(1);

// Modern assertions
expect(mockCallback).toHaveBeenCalledTimes(2);
expect(mockCallback).toHaveBeenCalledWith(0);
expect(mockCallback).toHaveBeenLastCalledWith(1);
expect(mockCallback).toHaveReturnedWith(43); // Last return value

// Detailed inspection still available
console.log(mockCallback.mock.calls); // All calls: [[0], [1]]
console.log(mockCallback.mock.results); // All results
  • Create a blank mock function.
  • Track calls, arguments, and return values.

Controlling Mock Behavior

// Modern mock controls
const mockFn = jest.fn();

// Example 1: Return values
mockFn.mockReturnValueOnce(true).mockReturnValue(false);
expect(mockFn()).toBe(true);
expect(mockFn()).toBe(false);

// Example 2: For async functions
const getUser = jest.fn()
  .mockResolvedValueOnce({ id: 1, name: 'Jane' })
  .mockResolvedValue({ id: 1, name: 'John' });

await expect(getUser()).resolves.toEqual({ id: 1, name: 'Jane' });
await expect(getUser()).resolves.toEqual({ id: 1, name: 'John' });

// Example 3: Rejecting promises
const failingFn = jest.fn().mockRejectedValue(new Error('API error'));
await expect(failingFn()).rejects.toThrow('API error');
  • `mockReturnValue(value)`: Sets a return value.
  • `mockResolvedValue(value)`: Sets a resolved promise value.
  • `mockImplementation(fn)`: Provides a custom implementation.

`jest.mock()`: Mocking Modules

Scenario: Testing a service that uses `axios` to fetch data.

// src/userService.js
import axios from 'axios';

export const getUser = async (id) => {
  const response = await axios.get(`https://api.example.com/users/${id}`);
  return response.data;
};

export const createUser = async (userData) => {
  const response = await axios.post('https://api.example.com/users', userData);
  return response.data;
};

// __tests__/userService.test.js
import { getUser, createUser } from '../src/userService';
import axios from 'axios';

// Mock the entire axios module
jest.mock('axios');

describe('userService', () => {
  test('should fetch user data', async () => {
    // Setup the mock return value
    axios.get.mockResolvedValue({
      data: { id: 1, name: 'Test User' }
    });

    const user = await getUser(1);
    
    expect(user.name).toBe('Test User');
    expect(axios.get).toHaveBeenCalledWith('https://api.example.com/users/1');
  });
  
  test('should create a new user', async () => {
    const newUser = { name: 'New User' };
    const createdUser = { id: 2, name: 'New User' };
    
    axios.post.mockResolvedValue({ data: createdUser });
    
    const result = await createUser(newUser);
    
    expect(result).toEqual(createdUser);
    expect(axios.post).toHaveBeenCalledWith(
      'https://api.example.com/users', 
      newUser
    );
  });
});
  • Mocks entire modules (e.g., `axios`, `fs`, database clients).
  • Prevents actual external calls during unit tests.

Partial Module Mocking

// For modules where you want to keep some real functionality
import fs from 'fs';

// Mock specific methods, keep others real
jest.mock('fs', () => {
  const originalModule = jest.requireActual('fs');
  
  return {
    ...originalModule,
    // Override only specific methods
    readFileSync: jest.fn().mockReturnValue('mocked content'),
  };
});

test('reads file with mock', () => {
  const content = fs.readFileSync('someFile.txt');
  expect(content).toBe('mocked content');
  expect(fs.readFileSync).toHaveBeenCalledWith('someFile.txt');
  
  // Other fs methods still work normally
  expect(typeof fs.writeFileSync).toBe('function');
});

Use `jest.requireActual()` to create partial mocks when you need to preserve some real behavior.

`jest.spyOn()`: Spying on Existing Methods

const calculator = {
  add: (a, b) => a + b,
  subtract: (a, b) => a - b,
};

describe('calculator', () => {
  test('add method should be called', () => {
    // Modern approach using spyOn
    const addSpy = jest.spyOn(calculator, 'add');
    
    const result = calculator.add(2, 3);
    
    expect(result).toBe(5); // Real implementation still works
    expect(addSpy).toHaveBeenCalledWith(2, 3);
    expect(addSpy).toHaveBeenCalledTimes(1);
    
    addSpy.mockRestore(); // Clean up spy
  });
  
  test('can override implementation temporarily', () => {
    const subtractSpy = jest.spyOn(calculator, 'subtract')
      .mockImplementation(() => 42); // Override implementation
    
    expect(calculator.subtract(10, 5)).toBe(42); // Returns mock value
    
    subtractSpy.mockRestore(); // Restore original
    expect(calculator.subtract(10, 5)).toBe(5); // Original works again
  });
});
  • Observes calls to existing methods without changing their implementation by default.
  • Can also temporarily override implementations.
  • `mockRestore()` returns to the original implementation.

Best Practices for Writing Good Tests

  • Readable & Descriptive: Clear names for `describe` blocks and `test` cases.
  • Arrange-Act-Assert (AAA):
    • Arrange: Set up test data/mocks.
    • Act: Execute the code under test.
    • Assert: Verify the outcome.
  • Test One Thing: Each test should ideally verify a single piece of functionality.
  • Independent & Repeatable: Tests should not rely on order or external state.
  • Don't Test Implementation Details: Test *behavior*, not internal mechanics.

Test Coverage: Measuring Our Progress

npx jest --coverage
  • Shows which lines/branches of code are executed by tests.
  • Helps identify untested areas.
  • Caution: High coverage != good tests. Still need meaningful assertions.
  • Aim for good coverage on critical business logic.

Integrating Tests into Your Workflow

  • Run on Save: Use `jest --watchAll` for instant feedback.
  • Pre-commit Hooks: Integrate with husky/lint-staged to run tests before commits.
  • CI/CD Pipelines: Automate test runs on every push to a repository (e.g., GitHub Actions, GitLab CI).
  • Code Reviews: Encourage reviewing tests alongside feature code.
